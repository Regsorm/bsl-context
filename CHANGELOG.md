Все заметные изменения этого проекта документируются здесь.

Формат: [Keep a Changelog](https://keepachangelog.com/ru/1.0.0/),
версионирование: [SemVer](https://semver.org/lang/ru/).

## [0.5.0] — 2026-07-09

### Добавлено

- **Новый инструмент `validate_module`** — валидация ЦЕЛОГО BSL-модуля.
  В отличие от `validate_expression`, принимающего фрагмент, здесь сервер
  через `tree-sitter-onescript` сам извлекает объявления `Процедура ИмяX(...)` /
  `Функция ИмяY(...)` и строит whitelist «своих» имён. Вызовы из whitelist в
  проверке глобальных вызовов пропускаются до fuzzy-этапа — вызов пользовательской
  процедуры больше не путается с опечаткой платформенного метода.
  Уровни/профили и формат ответа — как у `validate_expression`.

- **`unknown_global_method` теперь эмиттится по fuzzy-сходству** (раньше
  неизвестный глобальный вызов молча пропускался). При промахе прямого lookup в
  `PlatformIndex.global_methods` включается сверка через расстояние Левенштейна
  с двухпороговой эвристикой:
  - сильное сходство (distance ≤ 2 при len ≥ 5, либо distance ≤ 1 при len < 5) →
    Confidence::**High** («уверенная опечатка платформенного метода»);
  - слабое сходство (distance ≤ 3 при len ≥ 6) → Confidence::**Low**
    («возможная опечатка»);
  - далёкое — молчание (это внутренняя процедура, валидатору её не видно).
  Профиль `strict` фильтрует Low, слабые модели получают только High.

- **Новый вид находки `unknown_directive`** — валидация имён директив
  компилятора и расширений в `validate_module`. Whitelist: `НаКлиенте`,
  `НаСервере`, `НаСервереБезКонтекста`, `НаКлиентеНаСервере`,
  `НаКлиентеНаСервереБезКонтекста`, `Перед`, `После`, `Вместо`,
  `ИзменениеИКонтроль` (плюс их английские эквиваленты). Проверка при обходе
  `annotation`-узлов AST, тот же двухпороговый Confidence, что для
  `unknown_global_method` — `&НаКлентее` → suggestion `НаКлиенте`, High.

### Исправлено

- `check_global_calls` теперь не считает `&Перед("Foo")` за глобальный вызов
  `Перед(...)` — левый сосед `&` трактуется как annotation-директива, её
  валидация идёт отдельным путём через `walk_module_ast`. Иначе на любом модуле
  расширения был бы false-positive `UnknownGlobalMethod` на `Перед`/`После`/
  `Вместо`.

### Замечание

- Пороги fuzzy-эвристики (сильное/слабое сходство, минимальная длина имени)
  сейчас захардкожены в `expression::fuzzy_confidence_for`. Вынос в
  `config.toml` (`default_fuzzy_*`) отложен на следующую итерацию — при
  необходимости оператор может изменить их локальной сборкой.
- Валидация target-имени в override-директивах (`&Перед("ИмяЦелевойПроцедуры")`
  — существует ли эта процедура) не входит в 0.5.0: цель обычно объявлена в
  другом модуле конфигурации, за пределами анализируемого источника.

## [0.4.0] — 2026-05-21

### Добавлено

- **Уровень валидации 2.5 (`level=3`), под-фаза A — return-type tracking.**
  Тип локальной переменной теперь выводится не только из `Новый ТипX`,
  `ТипY.ЗначениеZ` и аннотации `// @type`, но и из **возвращаемого типа вызванного
  метода/свойства**:
  - `Х = Запрос.Выполнить()` → `Х: РезультатЗапроса` (по `Method.return_type` из hbk);
  - цепочки любой глубины: `Х = Запрос.Выполнить().Выбрать()` → `ВыборкаИзРезультатаЗапроса`
    (итеративный резолв звено-за-звеном);
  - свойства: `Х = Объект.Свойство` → тип из `Property.type_name`;
  - возврат глобального метода: `Х = ПолучитьОбщийМакет()` → `ТабличныйДокумент`.

  Дальше работает обычная проверка членов: опечатка в `Х.НетТакогоМетода()` ловится
  как `unknown_type_member` (confidence **Low** — зависит от точности type inference
  и полноты hbk). Резолвится только когда RHS — чистая цепочка (бинарные выражения,
  литералы, `Новый` — не выводятся, чтобы не плодить ложные типы).

- **Фильтр типов с динамическими членами** (применяется на всех уровнях). Для типов,
  чьи «члены» задаются в runtime и в hbk отсутствуют, проверка `Объект.Член` отключена:
  `ВыборкаИзРезультатаЗапроса`, `ВыборкаДанных`, `СтрокаТаблицыЗначений`,
  `СтрокаДереваЗначений` (колонки выборки/ТЗ/дерева); `Структура`, `ФиксированнаяСтруктура`
  (произвольные ключи); `Форма`, `УправляемаяФорма`, `ЭлементыФормы` (реквизиты/элементы
  конкретной формы — в её метаданных); `ОбъектXDTO`, `ЗначениеXDTO` (свойства XDTO).
  Без фильтра возникал массовый false-positive: return-type tracking порождал
  `Выборка.Регистратор`, `СтрокаТЗ.ОбъектОплаты`, а давняя проблема уровней 1/2 —
  `Форма.<реквизит>` (один тип `Форма` давал 36k ложных находок на конфигурации УТ).
  Регресс-прогон по 4 конфигурациям выявил это до релиза. Размен: не ловим опечатку
  в статическом методе такого типа, зато нет массового FP на динамических полях.

- Параметр `level=3` в `validate_expression` и `default_validation_level=3` в конфиге.
  Диапазон клампа расширен `[1..=2]` → `[1..=3]`. Дефолт по-прежнему `level=1`.
  Профиль `strict` форсирует `level=1` — новые Low-находки слабым моделям не приходят.

### Замечание

- Под-фаза B Уровня 2.5 (реквизиты справочников/документов через метаданные
  конфигурации, MCP `1c`) запланирована отдельной итерацией 0.5.0 — она требует
  HTTP-клиента и несёт основной риск false-positive (динамика менеджеров/ссылок,
  которых нет в hbk).
- Под-фаза A не имеет внешних зависимостей и не меняет поведение `level=1`/`level=2`
  (return-type-источник gated на `level >= 3`).

## [0.3.5] — 2026-05-21

### Изменено

- **Откат** глубокого фикса маскировки из v0.3.4 (врезки-комментарии `//` внутри
  многострочных строк). Регресс-прогон по 4 конфигурациям показал, что фикс
  нестабилен: на реальных данных дал **+193 новых false-positive `unknown_new_type`**
  (`Новый '_'`, `Новый 'отчёт'`, `Новый 'Тогда'`…) и разнонаправленный сдвиг
  `unknown_type_member` (−93k) — текст местами выходил из-под маскировки в обе
  стороны. `mask_strings_and_comments` возвращена к стабильной версии v0.3.3.
- `Число` (FP-коллизия с приведением типа `ЧИСЛО(n,m)` в языке запросов) —
  закрыта **точечным костылём** вместо фикса маскировки: имя `Число`/`Number`
  исключено из проверки числа аргументов (`is_query_cast_collision`). Риск нулевой,
  маскировка не тронута. Совместно с `ПродолжитьВызов` (v0.3.4) убирает основной
  остаток `wrong_argument_count`.

### Замечание
- Урок: глубокая правка маскировки рискованна (фундамент всех проверок);
  объективную оценку даёт только полный регресс-прогон по 4 конфигурациям —
  он и отсёк нестабильный фикс.

## [0.3.4] — 2026-05-21

### Исправлено

- Остаток false-positive `wrong_argument_count` (199 шт после v0.3.3), два класса:
  - **ПродолжитьВызов** (61) — спецконструкция расширений (`&Вместо`): число
    аргументов равно сигнатуре перехватываемого метода, т.е. произвольное.
    Добавлена в `is_variadic_global`.
  - **Число** (128) — `ЧИСЛО(n,m)` это приведение типа в языке запросов, а не
    вызов функции `Число()`. Корень: `mask_strings_and_comments` не понимала
    1С-особенность многострочных строк — между строками-продолжениями `|`
    допустимы врезки-комментарии `//`, и кавычки в таком комментарии ломали
    парность `"`, из-за чего часть текста запроса выходила из-под маскировки.
    Теперь внутри многострочного литерала строка, начинающаяся с `//`,
    маскируется как комментарий (кавычки в ней не считаются границами).
  - Остаток ~10 (затенение глобального имени пользовательской функцией,
    напр. `ПолучитьФайл`) — фундаментальное ограничение (нет парсинга функций
    конфигурации), оставлен как известный.

## [0.3.3] — 2026-05-21

### Исправлено

- False-positive `wrong_argument_count` на вариативных функциях платформы
  (`СтрШаблон`, `Макс`, `Мин`) — остаток после фикса v0.3.2 (~53k срабатываний).
  Два механизма в `brief_signature`:
  - **Диапазонный параметр** hbk вида `Значение1-Значение10` (СтрШаблон) — это
    один слот в сигнатуре, но синтаксически до 10 значений. Верхняя граница
    `max_args` расширяется по верхней цифре диапазона (`parse_range_upper`).
  - **Семантически вариативные** `Макс`/`Мин` — hbk даёт один параметр,
    вариативность только в тексте описания. Помечены флагом `variadic`
    (фиксированный список `is_variadic_global`), верхняя граница не проверяется.
  В `SignatureBrief` добавлено поле `variadic`; в сообщении допустимый диапазон
  вариативной функции рендерится как `N+`.

## [0.3.2] — 2026-05-21

### Исправлено

- Массовый false-positive `wrong_argument_count` на вызовах с единственным
  строковым аргументом: `НСтр("ru = '...'")`, `Тип("...")`,
  `ПредопределенноеЗначение("...")` и т.п. ложно помечались как «не принимает
  0 аргументов». Причина: `mask_strings_and_comments` стирает строковый литерал
  целиком (включая кавычки) в пробелы, а `count_top_level_args` трактовал «одни
  пробелы между скобок» как отсутствие аргументов. Теперь различаются реально
  пустые скобки `Метод()` (0 аргументов) и скобки с замаскированным аргументом
  `Метод("...")` (≥1 аргумент). Baseline 2026-05-21 показал ~918k срабатываний
  этого вида, из них ~560k — только на `НСтр` (карточка #1232).

## [0.3.1] — 2026-05-21

### Добавлено

- Поле конфига `allowed_hosts` — список разрешённых значений заголовка `Host`
  для `/mcp` (защита rmcp 1.6 от DNS-rebinding). По умолчанию — только loopback
  (`localhost`, `127.0.0.1`, `::1`).

### Исправлено

- При сетевом деплое (`host = "0.0.0.0"`) обращение к серверу по внешнему адресу
  возвращало `403 Forbidden: Host header is not allowed` — rmcp 1.6 по умолчанию
  принимает только loopback-Host. Теперь в `config.toml` можно добавить внешний
  адрес (например, `allowed_hosts = ["localhost","127.0.0.1","::1","<ip-сервера>"]`),
  и сервер становится доступен по сети. Запись без порта разрешает любой порт хоста.

## [0.3.0] — 2026-05-21

### Добавлено

- Поле `confidence` (`high` / `low`) у каждой находки `validate_expression`.
  Производно от `kind`: `high` (false-positive ≈ 0) — `unknown_enum_value`,
  `wrong_argument_count`; `low` (возможен false-positive) — `unknown_type_member`,
  `unknown_new_type`, `unknown_global_method`. Раньше маппинг «kind → надёжность»
  жил только во внешних правилах потребителя; теперь зашит в сам ответ
  (карточка-decision #1230).
- Параметр `profile` у `validate_expression`:
  - `"full"` (default) — все находки, `level` как передан. Для сильной модели,
    которая сама отбрасывает сомнительные срабатывания.
  - `"strict"` — только high-confidence находки и форсированный `level=1`.
    Для слабых моделей (LibreChat/DeepSeek): ложное срабатывание клиенту физически
    не приходит, нечем зацикливаться. Разблокирует безопасный доступ слабых
    агентов к валидатору.
- Поле конфига `default_profile` (`full` / `strict`) — профиль по умолчанию,
  если клиент не передал `profile`. Дефолт — `full`.

### Совместимость

- `validate_expression` без новых параметров ведёт себя как прежде, но каждая
  ошибка дополнительно содержит поле `confidence`. Параметры `level` и `profile`
  опциональны — обратная совместимость полная.

## [0.2.0] — 2026-05-18

### Изменено

- HTTP-транспорт MCP переведён в stateless-режим: `session::never::NeverSessionManager`
  + `StreamableHttpServerConfig::default().with_stateful_mode(false).with_json_response(true)`.
  Устраняет ошибку `404 Session not found`, возникавшую после рестарта сервера
  у клиентов с протухшим `Mcp-Session-Id` в памяти (claude-code VSCode extension
  2.1.141+ перестал делать auto-reinit на 404). Точная копия фикса, применённого
  в `mcp-cache-ci` v0.3.0 от 2026-05-14 (карточка #1184).

### Совместимость

- `POST /mcp` (initialize, tools/list, tools/call) — поведение идентично 0.1.x.
- `DELETE /mcp` (close session) — теперь `405 Method Not Allowed`.
- `GET /mcp` (standalone SSE stream) — теперь `405 Method Not Allowed`.
  Mainstream MCP-клиенты (VSCode `claude-code`, CLI `claude`, MCP SDK Python/TS,
  `mcp-cli`) эти методы не используют — совместимость с типовыми клиентами полная.

## [0.1.0] — 2026-05-06

- Первоначальный релиз. 9 MCP-tools контекста платформы 1С (типы, методы,
