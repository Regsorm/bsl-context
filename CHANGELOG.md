## [0.11.0] — 2026-07-20

**Валидатор научился читать язык запросов 1С и находить неоптимальные
конструкции. До этой версии текст запроса из анализа сознательно исключался —
маскировался вместе со строковыми литералами, чтобы не мешать разбору кода.**

Такой код корректен и проходит любую проверку синтаксиса, он просто медленный:
соединение с неиндексированной временной таблицей, чтение таблицы движений
вместо виртуальной, отсутствующий отбор по измерениям. Ни одна из шести находок
ниже не выдаётся другими инструментами: у BSL Language Server есть разбор SDBL,
но нет метаданных конфигурации.

### Добавлено

- **Крейт `sdbl-parse`** — разбор подмножества языка запросов (источники,
  соединения, временные таблицы, индексирование, условия). Готовой грамматики
  SDBL для Rust не существует: у `1c-syntax` она написана под ANTLR. Парсер
  рекурсивного спуска, без зависимостей. На конфигурации УТ разбирается **97.3%**
  из 23260 запросов, найденных в 14905 модулях.

- **Извлечение текста запроса из BSL** — `bsl_parse::collect_query_texts` с
  картой смещений `QueryText::map_offset`: находка встаёт на строку модуля, даже
  когда запрос собран из нескольких литералов. Символ продолжения `|`, отступ,
  комментарии между строками и удвоенные кавычки в значение не входят.

- **Шесть находок оптимальности запросов:**

  | Вид | confidence | Что ловит |
  |-----|-----------|-----------|
  | `temp_table_without_index` | high | `ПОМЕСТИТЬ` без `ИНДЕКСИРОВАТЬ ПО` при последующем соединении |
  | `or_in_join_condition` | high | `ИЛИ` разрывает условие связи |
  | `join_with_subquery` | low | Соединение с подзапросом вместо временной таблицы |
  | `physical_register_table` | low | Таблица движений регистра остатков вместо виртуальной |
  | `virtual_table_without_filter` | low | Виртуальная таблица без отбора по измерениям |
  | `join_on_unindexed_field` | low | У присоединяемой таблицы нет индекса по полю связи |

  Раскладка `confidence` выбрана по тому, чем оборачивается профиль `strict` у
  потребителя: он означает «блокировать сборку артефакта». Все шесть находок —
  про скорость, а не про корректность, поэтому блокируют только две, которые
  исправляются одной строкой и почти никогда не бывают лишними. Остальные
  четыре видны в профиле `full` и остаются рекомендацией: «взять виртуальную
  таблицу» может потребовать переписать половину запроса, а на малых объёмах
  выигрыша не даст вовсе — это решение автора кода.

- **Метод `object_schema` в контракте `SymbolSource`** — состав объекта:
  реквизиты, измерения, ресурсы, признак индексирования, вид регистра.
  Реализован во всех трёх источниках. Как и прежде, `None` означает «не знаю» и
  обязывает правило молчать целиком.

- **Схема 3 облегчённого индекса** (`lite-index`): таблица `object_fields` и
  колонка `objects.register_type`; XML объектов теперь разбирается по-настоящему
  (`quick-xml`), а не поиском подстроки. Состав СЛИВАЕТСЯ по всем копиям объекта:
  в выгрузке УТ у `Documents.ЗаказКлиента` 19 копий (базовая плюс расширения),
  полный состав только в одной. **Индексы схемы 2 требуют пересборки** —
  `rebuild_symbol_index`; до неё правила про состав молчат.

### Что показал замер на реальной конфигурации

Модульные тесты пропустили шесть дефектов, которые нашлись только прогоном по
корпусу УТ. Самые дорогие:

- Правило про неиндексированное поле давало **16360** находок, потому что
  требовало индекс у обеих сторон условия. Соединение ищет по индексу
  ПРИСОЕДИНЯЕМОЙ таблицы: в `… СОЕДИНЕНИЕ Справочник.Номенклатура КАК С
  ПО А.Номенклатура = С.Ссылка` левое поле индексировать не нужно. После
  исправления — **380**.
- Правило про физическую таблицу давало **1624** находки, пока не научилось
  различать запросы, которым нужны сами движения: регистратор, период,
  активность, а также реквизиты регистра (в виртуальных таблицах их нет вовсе).
  После — **34**.
- Правило про `ИЛИ` давало **589** находок, считая находкой `ИЛИ` внутри скобок
  рядом с равенствами — обычный дополнительный отбор, индексу он не мешает.
  Осталось **27**.

Итог на УТ: **3160 находок в 948 модулях из 14905**. Регресс правил 0.10.0 не
изменился — те же 219 находок `unknown_common_module` / `unknown_metadata_object`.

### Изменено

- `validate_module` описывает новые находки в своём `description` — модель
  узнаёт о них только оттуда.
- Корпусные тесты `corpus_config_objects` и новые `corpus_query_rules*` работают
  с индексом схемы 3 (`C:\Temp\ut_lite_v3.db`).
- README (обе версии), раздел «Чего валидатор не знает»: раньше он читался так,
  будто источник имён для `undeclared_method` бесполезен. На деле у находки три
  разные судьбы — метод глобального общего модуля и метод модуля
  объекта-владельца снимаются полностью, а «имя где-то в конфигурации есть, но
  видно ли отсюда — неизвестно» понижает уверенность до `low`. Не знает ни один
  источник только правила видимости. Заодно поправлено «workspace из пяти
  крейтов» — их девять.
- Комментарий к `SymbolSource::is_global_export` утверждал, что источники поверх
  code-index этого не умеют. Умеют все три: признак «Глобальный» читается из XML
  общего модуля.

## [0.10.0] — 2026-07-16

**Валидатор научился проверять существование объектов конфигурации. До этой
версии связь с конфигурацией работала в одну сторону — только чтобы ГАСИТЬ
ложные находки по методам; выдуманный общий модуль и выдуманный справочник
проходили как корректный код.**

Замер на восьми типовых ошибках: было 6 из 8 пойманных на `level=3`, стало
8 из 8; в профиле `strict` — было 3 из 8, стало 5 из 8. Оба ранее непойманных
случая — ровно самые частые ошибки слабой модели.

### Добавлено

- **Находка `unknown_common_module` (confidence `high`).** Обращение
  `ИмяМодуля.Метод(...)`, где общего модуля с таким именем в конфигурации нет:

  ```bsl
  УправлениеПрогрессом.Установить(1, 0);   // такого модуля нет
  ```

- **Находка `unknown_metadata_object` (confidence `high`).** Обращение
  `Менеджер.Имя` к менеджеру объектов конфигурации, где объекта с таким именем
  в коллекции нет:

  ```bsl
  С = Справочники.НесуществующийСправочник.ПустаяСсылка();
  ```

  Поддержаны 19 менеджеров (`Справочники`, `Документы`, `РегистрыСведений`,
  `Перечисления`, `ПланыВидовХарактеристик`, …). Обе находки работают на всех
  уровнях, включая `level=1`, и входят в профиль `strict`.

- **Контракт `SymbolSource` расширен** тремя методами с дефолтной реализацией
  (существующие источники не ломаются): `object_exists(collection, name)`,
  `collection_names(collection)` (для подсказки «возможно, вы имели в виду»),
  `global_variables()`. Все возвращают `Option`: `None` — «источник не знает»,
  и тогда валидатор ОБЯЗАН молчать.

- **`lite-index`: схема 2.** Таблица `objects` — имена объектов собираются по
  XML-файлам выгрузки (`<Коллекция>/<Имя>.xml`), а НЕ по таблице модулей: у
  объекта модуля может не быть вовсе (в УТ 909 из 1069 перечислений). Таблица
  `global_vars` — экспортные переменные модулей приложения. Индекс, собранный
  прежней версией, не ломается: отвечает «не знаю», проверка объектов на нём
  молчит до пересборки (`rebuild_symbol_index`).

- **`bsl-parse`:** переменные циклов (`loop_vars`) и признак «член — вызов, а не
  свойство» (`member_is_call`) в фактах разбора.

### Исправлено

- **`is_form_module` не узнавал каталог `CommonForms/`.** Общая форма — такой же
  модуль формы со своими реквизитами; из-за пропуска реквизит, затеняющий имя
  контекста (`Документы.Добавить()` в `CommonForms/ДокументыОснованияЭПД`, где
  `Документы` — таблица формы), давал ложную находку. Затрагивает и
  `shadowed_context_name`: замер на корпусе УТ показал, что число находок
  прочих видов не изменилось ни на единицу.

### Чего это стоило: пять причин ложных срабатываний

Регресс по 14905 модулям УТ — обязательный минимальный тест этого проекта —
дал на первом прогоне **40398 находок на заведомо рабочем коде**. Ни один
модульный тест их не показал. Итог после разбора: **219 находок, и все
настоящие** (объектов действительно нет в УТ; почти все — в модулях обмена
EnterpriseData/EDI и внешних обработках, написанных под семейство
конфигураций, где этот код мёртв).

| Причина | Ложных находок |
|---|---|
| Переменная цикла `Для Каждого` связывается циклом, а не присваиванием — в фактах разбора её не было | 39431 |
| `ПараметрыПриложения` — экспортная переменная модуля приложения, видна без префикса отовсюду | 123 |
| Методы менеджера (`ПланыОбмена.ГлавныйУзел()`) синтаксически неотличимы от имён объектов | 967 |
| Свойства платформенных перечислений и мусор восстановления дерева после ошибки (`"1.2.643"` как голова) | 9 |
| `is_form_module` не узнавал `CommonForms/` | 1 |

### Гочи источника поверх `code-index` (стоили ложных находок на живом коде)

Ошибки были НАШИ: `code-index` про обе обрезки честно сообщает — источник не
читал его пометки. Оба дефекта невидимы для `LiteSource`, стабов и корпусного
регресса, их ловит только запрос к живому серверу (тесты `live_code_index_source`):

- **Страж размера ответа.** `bsl_sql` входит в `DEFAULT_CAP_TOOLS` у
  `code-index`: ответ режется по `[mcp].max_response_bytes` (48 000 байт).
  Список общих модулей УТ весит ~170 КБ — приходило **386 строк из 3091**, по
  алфавиту до «И», и реальный `ОбщегоНазначения` объявлялся несуществующим.
  `code-index` при этом сообщал об обрезке четырьмя способами: `rows_total: 3091`,
  `rows_truncated: true`, `response_truncated: true` и подсказка
  `response_truncated_hint` с названием параметра конфига. Источник же читал
  единственное поле `truncated` — а оно про СОБСТВЕННЫЙ лимит `bsl_sql` и
  правдиво отвечало `false`. Забрать набор одним запросом нельзя в принципе —
  читаем страницами по 400 строк и проверяем все маркеры.
- **Сессионный дедуп.** `serve` считает отпечаток каждой СТРОКИ и опускает уже
  отданные в этой сессии, **явно помечая** это полем
  `rows_elided_already_delivered` (в `serve_dedup.rs` так и написано: «маркер
  вместо тишины… это correctness-sensitive, потому маркер явный»). Источник
  пометку не проверял. Имена пересекаются между коллекциями (`Закупки` — и общий
  модуль, и регистр накопления; `ПодарочныеСертификаты` — и регистр, и
  справочник), поэтому `SELECT name` терял их во втором запросе. Запрашиваем
  `full_name` — он уникален глобально, дедупу нечего опускать.

## [0.9.0] — 2026-07-13

**Находка класса ошибок, который молча пропускают и компилятор, и BSL LS:
переменная, чьё имя занято реквизитом контекста, доступным только для
чтения.**

### Добавлено

- **Находка `shadowed_context_name` (confidence всегда `high`).** Имя локальной
  переменной совпало со свойством контекста модуля, доступным ТОЛЬКО ДЛЯ
  ЧТЕНИЯ. Присваивание такому имени не создаёт переменную — оно падает в
  рантайме. Реальный инцидент 13.07.2026, модуль управляемой формы внешней
  обработки:

  ```bsl
  Параметры = Новый Структура;
  ```

  Ошибка рантайма: «Поле объекта недоступно для записи (Параметры)». Ни BSL LS,
  ни конфигуратор её не ловят.

  Ловится присваивание (`Имя = ...`) и объявление (`Перем Имя;`); обращение
  через точку (`Элементы.Список.Видимость = Ложь`) и вызов (`Закрыть()`) —
  не находка. Два правила:

  - свойства ГЛОБАЛЬНОГО контекста только для чтения (`Справочники`,
    `Документы`, `Метаданные`, …) — в любом модуле, кроме модуля формы;
  - свойства `ФормаКлиентскогоПриложения` только для чтения (`Параметры`,
    `Элементы`, `Команды`, `ЭтотОбъект`, …) — только в модуле управляемой
    формы; для этого правила обязателен параметр `module_path`.

  Имена берутся из справки платформы (`PlatformIndex`), а не из хардкод-списка:
  состав членов зависит от версии платформы. Модуль формы определяется по
  `module_path`, поддержаны две раскладки: `.../Forms/<Имя>/Ext/Form/Module.bsl`
  (конфигурация) и `.../Form/<Имя>/Form.obj.bsl` (внешняя обработка, v8unpack).

- **Инструмент `reserved_names` (без параметров).** Отдаёт JSON с четырьмя
  группами имён из справки той версии платформы, что задана в конфиге
  (`platform_path`), плюс `counts`. Присваивание падает в рантайме:
  `global_readonly` (98 имён — в любом модуле), `form_readonly` (13 — в модуле
  управляемой формы). Присваивание не падает, но локальная переменная НЕ
  создаётся — молча меняется настройка сеанса либо сама форма:
  `global_writable` (2 — `РабочаяДата`, `ГлавныйСтиль`), `form_writable`
  (37 — `Заголовок`, `Модифицированность`, …). Нужен для генерации кода: взять
  имя переменной, не пересекающееся с контекстом.

- **Параметр `form_attributes` у `validate_module`** — имена реквизитов формы.
  Реквизит перекрывает имя контекста (в УТ есть формы с реквизитами `Метаданные`,
  `БезопасноеХранилище`, `ПараметрЗапуска`), поэтому такие имена из проверки
  исключаются. Передав состав реквизитов, вы включаете внутри модуля формы и
  проверку имён глобального контекста: `Справочники = Новый Соответствие;` в
  форме без такого реквизита — ошибка. Без параметра эта проверка в формах
  выключена: не зная состава реквизитов, валидатор молчит, чтобы не соврать.

### Изменено

- **У `validate_module` параметр `module_path` теперь нужен ещё и для правила
  по членам формы** (раньше — только для экспортных методов модуля
  объекта-владельца).

### Замер на УТ

На 14905 модулях УТ 11.5 наивная проверка (без исключений) давала 5706
находок, почти все ложные. После всех условий осталась 1 находка, и она
настоящая — `Параметры = Новый Структура;` в функции
`ПараметрыОбработкиОшибкиОтправки()` внешней обработки «Ручная отправка
писем» (падает при обработке ошибки отправки письма). Что снимает шум:

- имя связано локально — параметр процедуры (`Процедура
  Обработчик(УчетнаяЗаписьНастроена, Параметры)` — так пишет сама 1С) либо
  объявленная `Перем` переменная;
- процедура «БезКонтекста» (`&НаСервереБезКонтекста`,
  `&НаКлиентеНаСервереБезКонтекста`): контекста формы там нет,
  `Элементы = Форма.Элементы;` — идиома БСП (319 мест в УТ);
- модуль обычной (неуправляемой) формы — другой тип контекста; распознаётся
  по отсутствию директив компиляции в модуле (30 мест в УТ);
- свойство, доступное для записи (`Заголовок`, `Модифицированность`,
  `ТолькоПросмотр`) — присваивание работает и штатно меняет форму (5294
  места в УТ);
- имя метода формы (`Закрыть`) — реквизит формы можно назвать как метод,
  платформа разводит данные и вызов (15 мест в УТ);
- правило по глобальному контексту выключено внутри модулей форм: реквизит
  формы может называться как глобальное свойство (в УТ есть формы с
  реквизитами `Метаданные`, `БезопасноеХранилище`, `ПараметрЗапуска`), а
  состава реквизитов формы валидатор не видит — ему дают только текст
  модуля.

Валидатору не виден состав реквизитов формы вовсе (в т.ч. основной реквизит
`Объект`) — их нет в справке платформы.

Ломающих изменений API нет.

## [0.8.0] — 2026-07-13

**Один сервер обслуживает несколько конфигураций 1С: у каждой свой источник имён и
свой способ доступа к нему. И источник больше не может молча врать.**

### Добавлено

- **Несколько конфигураций на одном сервере (секция-список `[[symbol_sources]]`).**
  У каждой записи свой алиас (`repo`) и свой способ доступа: собственный индекс
  (`lite`), прямое чтение базы `code-index` (`code_index_db`) или `code-index` по сети
  (`code_index_mcp`). Смешивать способы можно свободно.

  У `validate_module` и `rebuild_symbol_index` появился параметр `repo` — тот самый
  алиас. Остальных восьми инструментов это не касается: они отвечают из справки
  платформы и про конфигурации ничего не знают.

- **Необязательное поле `code_index_repo`** — на случай, когда в `code-index`
  репозиторий назван иначе, чем конфигурация в bsl-context. Не задано — берётся `repo`.

- **Проверка репозитория при подключении к `code-index`.** Сервер спрашивает
  статистику по своему репозиторию (14 мс) и, если `code-index` его не знает, источник
  не создаётся, а в журнал идёт ошибка со списком доступных имён.

- **Признак «источник нездоров».** Ошибка обращения к `code-index` помечает источник
  нерабочим; пустой ответ больше не попадает в кэш.

### Изменено

- **Отказ вместо заведомо ложных находок.** Если конфигурация объявлена, но её источник
  имён не поднялся (или отвалился на ходу), `validate_module` по этому алиасу отказывает
  и объясняет причину. Раньше он молча валидировал без имён — а это на УТ давало 1420
  находок «метод не объявлен» высокой уверенности вместо 44.

- **`/health`: поле `symbol_source` заменено на `symbol_sources`** — карта «алиас →
  состояние источника».

### Несовместимые изменения

- **`repo` обязателен**, если на сервере настроена хотя бы одна конфигурация — даже если
  она единственная. Вызов `validate_module` без него отвергается с перечислением
  доступных алиасов. Причина: вызов должен быть однозначным, а молчаливая подстановка
  единственной конфигурации ломается ровно в тот момент, когда добавляют вторую.
  Конфигураций не настроено вовсе — параметр не нужен, код проверяется только против
  справки платформы.

- Секции `[symbol_source]` и `[[symbol_sources]]` взаимоисключающи. Старая одиночная
  секция продолжает работать: её `repo` становится алиасом конфигурации, а без него
  берётся алиас `default`.

## [0.7.0] — 2026-07-10

**Валидатор узнал про остальную конфигурацию: вызов процедуры из чужого модуля
больше не считается опиской. Плюс белый список инструментов и пересборка индекса
имён прямо из MCP.**

Замеры — на 14905 модулях УТ (`C:\RepoUT-test`) и 18297 модулях ТДК (`C:\RepoBP_TDK`).

### Добавлено

- **Внешний источник имён конфигурации (секция `[symbol_source]`).** Валидатору
  присылают текст одного модуля, поэтому вызов процедуры, объявленной в другом
  файле, он принимал за описку. Источник отвечает на три вопроса: объявлен ли
  где-нибудь такой метод, экспортный ли это метод глобального общего модуля, и
  какие экспортные методы у модуля объекта-владельца внешней обработки.

  На УТ: **1420 находок `undeclared_method` высокой уверенности → 44**, из них
  настоящая одна (`глЗначениеПеременной` — функция из УТ 10.3, в УТ 11 её нет).
  На ТДК: 6 → 1 (`ЦифровойСлед()` зовёт `ПолучитьИнформациюОСетевыхАдаптерах`,
  которого нет ни в конфигурации, ни в платформе 8.3.27). Прочие виды находок
  не изменились ни на единицу.

  Три способа подключения дают одинаковый результат:

  - `kind = "lite"` — собственный облегчённый индекс (крейт `lite-index`);
    сервер ни от чего не зависит;
  - `kind = "code_index_db"` — прямое чтение базы `code-index` на этой же машине;
  - `kind = "code_index_mcp"` — через запущенный сервис `code-index`, машина любая.

  Признак «глобальный общий модуль» источники поверх `code-index` берут из XML
  модуля, который тот хранит целиком: отдельной колонки для этого не нужно.

- **Инструмент `rebuild_symbol_index`.** Пересобирает собственный индекс имён.
  Параметров не принимает — `root` и `db_path` берутся из конфига, поэтому клиент
  не может заставить сервер писать файл куда угодно. Каталог для базы создаётся
  сам. Сборка идёт во временный файл и подменяет старую базу целиком: если она
  упадёт, рабочая база останется прежней. Во время сборки валидация продолжает
  отвечать по прежнему индексу.

- **Белый список инструментов (секция `[tools]`).** `enabled = ["validate_module"]`
  оставит из десяти инструментов только валидацию. Скрытые не попадают в
  `tools/list` и отбиваются при прямом вызове. Пустой список или отсутствие секции —
  доступны все. Опечатка в имени сервер не роняет: предупреждение в лог.

- **Параметр `module_path` у `validate_module`.** Относительный путь модуля в
  выгрузке. Нужен, чтобы учесть экспортные методы модуля объекта-владельца, когда
  проверяется модуль обычной формы внешней обработки.

### Изменено

- `/health` показывает подключённый источник имён и читает его состояние на каждый
  запрос — после `rebuild_symbol_index` строка обновляется.
- Отсутствие файла базы при `kind = "lite"` больше не ошибка: в лог уходит
  предупреждение с подсказкой вызвать `rebuild_symbol_index`. Это штатный первый запуск.

### Исправлено

- **Конструктор с директивой препроцессора внутри аргументов.** `Новый Структура("а,б",
  Новый ОписаниеТипов(...), #Если ВебКлиент Тогда … )` разваливал разбор, и конструктор
  приходил обычным вызовом — 35 ложных находок на УТ. Теперь слово `Новый` слева от
  вызова распознаётся отдельно, на границе слова и без учёта регистра.
- **Модуль формы внешней обработки, которой ещё нет в индексе.** Владелец теперь
  выводится из пути формы, а не ищется в базе: новая форма, не попавшая в индекс,
  больше не даёт ложного предупреждения на законном вызове метода своего объекта.

### Добавлены крейты

- `lite-index` — облегчённый индекс конфигурации: объявления процедур и функций,
  флаг глобального общего модуля, владелец формы внешней обработки. Сборка УТ —
  12.5 с, 82 МБ. Дублирует часть данных `code-index` сознательно: `bsl-context`
  публикуется как самостоятельный сервер, и требовать второй продукт нельзя.
- `symbol-source` — три реализации источника имён.
- `bsl-parse` — общий слой разбора BSL на tree-sitter.

## [0.6.0] — 2026-07-10

**Слияние двух инструментов валидации в один, строгая проверка необъявленных
вызовов, смена грамматики BSL и починка маскировки строк.**

Все цифры ниже — замер на 14905 модулях конфигурации УТ (`C:\RepoUT-test`).

### Добавлено

- **Находка `undeclared_method` (confidence `high`).** При проверке ЦЕЛОГО модуля
  вызов `Имя(...)`, которого нет ни среди объявлений этого текста, ни среди
  платформенных методов, и который ни на что не похож (fuzzy промолчал), теперь
  считается опиской либо забытой процедурой. Раньше такой вызов молча
  пропускался.

  Чтобы находка не превратилась в шум, перед ней стоят отсечки:
  вызов после `Новый` (это конструктор, а не метод); ключевые слова языка
  (`Пока`, `ВызватьИсключение`, `Попытка`, …, добавлены в список); имя, которое
  является методом ЛЮБОГО платформенного типа — внутри собственного модуля формы
  или объекта его методы зовутся без префикса (`Закрыть()`, `ЭтоНовый()`,
  `РеквизитФормыВЗначение(...)`), это не глобальные методы, и `find_global_method`
  их не видит.

  Отдельно выключается строгий режим для **модуля расширения** (есть директива
  `&Перед`/`&После`/`&Вместо`/`&ИзменениеИКонтроль`): такой модуль компилируется
  вместе с расширяемым и напрямую зовёт его процедуры, текста которых у
  валидатора нет. Без этой отсечки одно только `ДобавитьПКС` (объявлено в базовом
  типовом модуле на 65097-й строке) давало 2472 ложных находки.

- `PlatformIndex::all_type_method_names()` — имена методов всех типов платформы,
  считаются один раз и кэшируются (`OnceLock`).

### Изменено

- **BREAKING:** инструмент `validate_expression` удалён из MCP-набора; его роль
  полностью покрывает `validate_module`, который теперь принимает и целый
  модуль, и произвольный фрагмент (у фрагмента набор объявленных процедур
  пуст, поведение совпадает с прежним `validate_expression`, плюс проверка
  директив). Параметр `source` принимает прежние псевдонимы `bslSnippet` /
  `snippet` / `code` / `bslModule` / `module`, поэтому старые вызовы клиентов
  продолжают работать — менять надо только имя инструмента. Мотив: два
  инструмента с почти одинаковым описанием заставляли модель выбирать, и она
  регулярно выбирала не тот. Публичный API крейта `bsl-validator`
  (`validate_expression`, `validate_expression_at_level`,
  `validate_expression_with_profile`) сохранён.

- **Грамматика BSL: `tree-sitter-onescript` → `tree-sitter-bsl` 0.1.7.** Старая
  не разбирала двойную индексацию (`Знач["a"]["b"]`, `Ф(Х)["a"]["b"]`): дерево
  уходило в `ERROR`, и объявления процедур после такого места терялись — а
  строгая проверка вызовов на неполном списке объявлений даёт шквал ложных
  находок. Сравнение на 14905 модулях:

  | Показатель | onescript | bsl |
  |---|---|---|
  | файлов с `ERROR` | 1328 | 208 |
  | потеряно объявлений | 1684 | 34 |

  Смена грамматики убрала 5506 ложных находок (12194 → 6688).

- **Объявления собираются из ДВУХ источников: дерева и текста.** Ни один не
  полон: разбор находит 2753 имени, невидимых построчному проходу (заголовок с
  переносом строки перед скобкой), а текст спасает 34 имени, теряемых разбором на
  файлах с `ERROR`. Берётся объединение (`walk_module_ast` ∪ `scan_declarations`).

- **Проверка директив переехала с AST на текст.** `tree-sitter-bsl` заводит узел
  `annotation` только для директив, которые знает сама грамматика, а неизвестная —
  то есть ровно та, ради которой проверка и существует (`&НаКлентее`) — попадает в
  `ERROR` без выделенного имени. Теперь `scan_directives` читает замаскированный
  текст.

### Исправлено

- Ложный `High` на именах с приставкой. Правило «`len >= 5` и `distance <= 2`»
  считало опечаткой имя, к которому спереди приписаны символы: `тФормат` →
  предлагал `Формат`, `тзСтрока` и `ТЗСтрока` → `Строка`. Предохранитель 0.5.1
  ловил только приписанный СУФФИКС (`СтрокаТЧ`, `Формат1`), потому что
  проверял, является ли подсказка началом имени. Добавлен зеркальный
  `strip_suffix_ci`: если подсказка — строгий конец имени, находка подавляется
  при приставке из цифр, из 2+ символов, либо из одной строчной буквы перед
  заглавной (венгерская нотация). Удвоение первой буквы (`ССообщить` →
  `Сообщить`, `ФФормат` → `Формат`) и перестановки (`СтрНайит` → `СтрНайти`)
  по-прежнему ловятся с `High`.

- **Маскировка строк: комментарий внутри многострочного литерала.** Платформа
  разрешает вставлять `//`-комментарии между строками-продолжениями (`|`)
  строкового литерала. Если в закомментированной строке оставалась закрывающая
  кавычка (обычное дело: так выключают условие внутри текста запроса), маскировка
  считала литерал закрытым не там, и весь текст ниже инвертировался — код
  принимался за строку, а текст запроса за код. В находках появлялись `СУММА`,
  `ВЫРАЗИТЬ`, `ЕСТЬNULL`, `МАКСИМУМ`, `В` — ключевые слова языка запросов.

- **Маскировка строк: блоки `#Удаление` в модуле расширения.** Код между
  `#Удаление` и `#КонецУдаления` в скомпилированный модуль не попадает, но может
  обрывать строковый литерал на середине — тогда файл сам по себе не является
  корректным BSL. Новый `strip_extension_directives` затирает такие блоки и
  строки-маркеры (`#Вставка`, `#КонецВставки`, …) пробелами до всякого разбора,
  сохраняя номера строк и байтовые позиции.

  Обе правки вместе убрали остаток утечек: на 14905 модулях слова языка запросов
  больше не доживают до проверки ни разу.

### Что валидатор принципиально не может знать

Строгая проверка `undeclared_method` опирается на текст ОДНОГО модуля. Разбор
этого текста доведён до безошибочности: на 14905 модулях УТ **не осталось ни
одной находки, чьё объявление лежало бы в этом же файле**. Оставшиеся 1658
находок (в 131 модуле из 14905) вызваны не разбором, а тем, что имя объявлено в
ДРУГОМ файле конфигурации:

- **Модуль обычной (неуправляемой) формы** видит экспортные методы модуля объекта
  своего владельца и зовёт их без префикса. Пример: `external/Контур EDI` — формы
  зовут `ВывестиПредупреждение_КонтурEDI`, объявленную в
  `ExternalDataProcessor.obj.bsl`. Это 1374 находки из 1658.
- **Модуль расширения** видит процедуры расширяемого модуля (отсекается по
  директивам подключения; остаток — расширения без таких директив).
- **Глобальный общий модуль** (`<Global>true</Global>`) — его процедуры зовутся
  без префикса из любого модуля.

Во всех трёх случаях нужного текста в запросе просто нет, а флаг «глобальный»
лежит в метаданных конфигурации, которых у платформенного индекса нет по
определению. Устранить это чтением текста присланного модуля невозможно.

## [0.5.0] — 2026-07-09

### Добавлено

- **Новый инструмент `validate_module`** — валидация ЦЕЛОГО BSL-модуля.
  В отличие от `validate_expression`, принимающего фрагмент, здесь сервер
  через `tree-sitter-onescript` сам извлекает объявления `Процедура ИмяX(...)` /
  `Функция ИмяY(...)` и строит whitelist «своих» имён. Вызовы из whitelist в
  проверке глобальных вызовов пропускаются до fuzzy-этапа — вызов пользовательской
  процедуры больше не путается с опечаткой платформенного метода.
  Уровни/профили и формат ответа — как у `validate_expression`.

- **`unknown_global_method` теперь эмиттится по fuzzy-сходству** (раньше
  неизвестный глобальный вызов молча пропускался). При промахе прямого lookup в
  `PlatformIndex.global_methods` включается сверка через расстояние Левенштейна
  с двухпороговой эвристикой:
  - сильное сходство (distance ≤ 2 при len ≥ 5, либо distance ≤ 1 при len < 5) →
    Confidence::**High** («уверенная опечатка платформенного метода»);
  - слабое сходство (distance ≤ 3 при len ≥ 6) → Confidence::**Low**
    («возможная опечатка»);
  - далёкое — молчание (это внутренняя процедура, валидатору её не видно).
  Профиль `strict` фильтрует Low, слабые модели получают только High.

- **Новый вид находки `unknown_directive`** — валидация имён директив
  компилятора и расширений в `validate_module`. Whitelist: `НаКлиенте`,
  `НаСервере`, `НаСервереБезКонтекста`, `НаКлиентеНаСервере`,
  `НаКлиентеНаСервереБезКонтекста`, `Перед`, `После`, `Вместо`,
  `ИзменениеИКонтроль` (плюс их английские эквиваленты). Проверка при обходе
  `annotation`-узлов AST, тот же двухпороговый Confidence, что для
  `unknown_global_method` — `&НаКлентее` → suggestion `НаКлиенте`, High.

### Исправлено

- `check_global_calls` теперь не считает `&Перед("Foo")` за глобальный вызов
  `Перед(...)` — левый сосед `&` трактуется как annotation-директива, её
  валидация идёт отдельным путём через `walk_module_ast`. Иначе на любом модуле
  расширения был бы false-positive `UnknownGlobalMethod` на `Перед`/`После`/
  `Вместо`.

### Замечание

- Пороги fuzzy-эвристики (сильное/слабое сходство, минимальная длина имени)
  сейчас захардкожены в `expression::fuzzy_confidence_for`. Вынос в
  `config.toml` (`default_fuzzy_*`) отложен на следующую итерацию — при
  необходимости оператор может изменить их локальной сборкой.
- Валидация target-имени в override-директивах (`&Перед("ИмяЦелевойПроцедуры")`
  — существует ли эта процедура) не входит в 0.5.0: цель обычно объявлена в
  другом модуле конфигурации, за пределами анализируемого источника.

## [0.4.0] — 2026-05-21

### Добавлено

- **Уровень валидации 2.5 (`level=3`), под-фаза A — return-type tracking.**
  Тип локальной переменной теперь выводится не только из `Новый ТипX`,
  `ТипY.ЗначениеZ` и аннотации `// @type`, но и из **возвращаемого типа вызванного
  метода/свойства**:
  - `Х = Запрос.Выполнить()` → `Х: РезультатЗапроса` (по `Method.return_type` из hbk);
  - цепочки любой глубины: `Х = Запрос.Выполнить().Выбрать()` → `ВыборкаИзРезультатаЗапроса`
    (итеративный резолв звено-за-звеном);
  - свойства: `Х = Объект.Свойство` → тип из `Property.type_name`;
  - возврат глобального метода: `Х = ПолучитьОбщийМакет()` → `ТабличныйДокумент`.

  Дальше работает обычная проверка членов: опечатка в `Х.НетТакогоМетода()` ловится
  как `unknown_type_member` (confidence **Low** — зависит от точности type inference
  и полноты hbk). Резолвится только когда RHS — чистая цепочка (бинарные выражения,
  литералы, `Новый` — не выводятся, чтобы не плодить ложные типы).

- **Фильтр типов с динамическими членами** (применяется на всех уровнях). Для типов,
  чьи «члены» задаются в runtime и в hbk отсутствуют, проверка `Объект.Член` отключена:
  `ВыборкаИзРезультатаЗапроса`, `ВыборкаДанных`, `СтрокаТаблицыЗначений`,
  `СтрокаДереваЗначений` (колонки выборки/ТЗ/дерева); `Структура`, `ФиксированнаяСтруктура`
  (произвольные ключи); `Форма`, `УправляемаяФорма`, `ЭлементыФормы` (реквизиты/элементы
  конкретной формы — в её метаданных); `ОбъектXDTO`, `ЗначениеXDTO` (свойства XDTO).
  Без фильтра возникал массовый false-positive: return-type tracking порождал
  `Выборка.Регистратор`, `СтрокаТЗ.ОбъектОплаты`, а давняя проблема уровней 1/2 —
  `Форма.<реквизит>` (один тип `Форма` давал 36k ложных находок на конфигурации УТ).
  Регресс-прогон по 4 конфигурациям выявил это до релиза. Размен: не ловим опечатку
  в статическом методе такого типа, зато нет массового FP на динамических полях.

- Параметр `level=3` в `validate_expression` и `default_validation_level=3` в конфиге.
  Диапазон клампа расширен `[1..=2]` → `[1..=3]`. Дефолт по-прежнему `level=1`.
  Профиль `strict` форсирует `level=1` — новые Low-находки слабым моделям не приходят.

### Замечание

- Под-фаза B Уровня 2.5 (реквизиты справочников/документов через метаданные
  конфигурации, MCP `1c`) запланирована отдельной итерацией 0.5.0 — она требует
  HTTP-клиента и несёт основной риск false-positive (динамика менеджеров/ссылок,
  которых нет в hbk).
- Под-фаза A не имеет внешних зависимостей и не меняет поведение `level=1`/`level=2`
  (return-type-источник gated на `level >= 3`).

## [0.3.4] — 2026-05-21

### Исправлено

- Остаток false-positive `wrong_argument_count` (199 шт после v0.3.3), два класса:

### Исправлено

## [0.3.2] — 2026-05-21

### Исправлено

- Массовый false-positive `wrong_argument_count` на вызовах с единственным
  строковым аргументом: `НСтр("ru = '...'")`, `Тип("...")`,
  `ПредопределенноеЗначение("...")` и т.п. ложно помечались как «не принимает

  этого вида, из них ~560k — только на `НСтр` (карточка #1232).

## [0.3.0] — 2026-05-21

### Добавлено

- Поле `confidence` (`high` / `low`) у каждой находки `validate_expression`.
  Производно от `kind`: `high` (false-positive ≈ 0) — `unknown_enum_value`,
  `wrong_argument_count`; `low` (возможен false-positive) — `unknown_type_member`,
  `unknown_new_type`, `unknown_global_method`. Раньше маппинг «kind → надёжность»
  жил только во внешних правилах потребителя; теперь зашит в сам ответ
  (карточка-decision #1230).
- Параметр `profile` у `validate_expression`:
  - `"full"` (default) — все находки, `level` как передан. Для сильной модели,
    которая сама отбрасывает сомнительные срабатывания.
  - `"strict"` — только high-confidence находки и форсированный `level=1`.
    Для слабых моделей (LibreChat/DeepSeek): ложное срабатывание клиенту физически
    не приходит, нечем зацикливаться. Разблокирует безопасный доступ слабых
    агентов к валидатору.
- Поле конфига `default_profile` (`full` / `strict`) — профиль по умолчанию,
  если клиент не передал `profile`. Дефолт — `full`.

### Совместимость

- `validate_expression` без новых параметров ведёт себя как прежде, но каждая
  ошибка дополнительно содержит поле `confidence`. Параметры `level` и `profile`
  опциональны — обратная совместимость полная.
