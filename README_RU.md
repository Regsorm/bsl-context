Опубликовано на Инфостарте: [bsl-context — проверка ИИ-кода 1С на соответствие API платформы](https://infostart.ru/1c/articles/2698363/)

MCP-сервер контекста платформы **1С:Предприятие 8.3**: типы, методы, свойства,
конструкторы, значения системных перечислений — плюс статическая валидация
BSL-выражений против реального индекса платформы.

Источник данных — синтакс-помощник платформы (`shcntx_ru.hbk`), который сервер
парсит собственным ридером (запуск 1С при этом не требуется).

## Зачем

Языковые модели и линтеры хорошо ловят синтаксис BSL, но «слепы» к ссылочной
корректности относительно платформы: существует ли значение системного
перечисления, есть ли у платформенного типа нужный метод, попадает ли число
аргументов глобальной функции в её перегрузки. `bsl-context` закрывает именно
этот слой — сверяет код с фактическим составом API конкретной версии платформы.

## Возможности

**Справочные инструменты** — поиск и детали по типам, методам, свойствам,
конструкторам и значениям перечислений платформы.

**Валидация BSL-кода** (`validate_module`) — принимает и целый текст модуля
(общий модуль, модуль объекта, модуль формы), и произвольный фрагмент. Сервер
через `tree-sitter-bsl` сам извлекает из присланного текста объявления
`Процедура`/`Функция` и не считает их вызовы опечатками платформенных методов
(у фрагмента этот набор просто пуст). Дополнительно проверяет имена директив
компилятора и расширений (`&НаСервере`, `&Перед("Foo")`, …). Возвращает список
находок с указанием строки, колонки, вида и уверенности (`confidence`):

| Вид находки | confidence | Что значит |
|-------------|-----------|------------|
| `unknown_enum_value` | high | Значение системного перечисления не существует |
| `wrong_argument_count` | high | Число аргументов глобальной функции вне её перегрузок |
| `unknown_type_member` | low | У платформенного типа нет такого метода/свойства |
| `unknown_new_type` | low | Конструктор `Новый ТипX` неизвестен платформе |
| `unknown_global_method` | high / low | Неизвестная глобальная функция, похожая на платформенную (fuzzy: сильное сходство → high, слабое → low) |
| `undeclared_method` | high | Вызов не объявлен в присланном модуле и не найден в платформе (только при проверке целого модуля; в модуле расширения не выдаётся) |
| `unknown_directive` | high / low | Имя директивы (`&НаСервере`, `&Перед`, …) не входит в whitelist |

high-confidence находки имеют false-positive близкий к нулю; low — зависят от
точности вывода типов и полноты `hbk`.

### Уровни валидации

Глубина анализа задаётся параметром `level` (или `default_validation_level` в
конфиге), клампится в `[1..=3]`:

- **1** — ссылки с явным именем типа в исходнике (`Новый ТипX`, `ТипY.ЗначениеZ`,
  число аргументов глобальных функций). Низкий шум, безопасный дефолт.
- **2** — дополнительно локальный type inference в пределах процедуры:
  `Х = Новый ТипX`, `Х = ТипY.ЗначениеZ`, аннотация `// @type ТипX`.
- **3** — дополнительно return-type tracking: тип переменной из возвращаемого
  типа метода/свойства, включая цепочки `Запрос.Выполнить().Выбрать()`.

Чем выше уровень — тем больше находок и потенциальных false-positive.

### Чего валидатор не знает

Валидатор видит текст ОДНОГО модуля и платформенный контекст. Он не знает
метаданных конфигурации, поэтому не может проверить существование прикладных
процедур, объявленных в других модулях. Находка `undeclared_method`
не выдаётся, когда имя может прийти извне:

- модуль обычной (неуправляемой) формы зовёт экспортные методы модуля объекта своего владельца без префикса;
- модуль расширения зовёт процедуры расширяемого модуля (такой модуль распознаётся по директивам `&Перед`, `&После`, `&Вместо`, `&ИзменениеИКонтроль`, и строгая проверка для него выключается);
- глобальный общий модуль (`Глобальный = Истина`) зовётся без префикса из любого модуля.

Первый и третий случаи по тексту модуля не распознаются — там возможны ложные
срабатывания `undeclared_method`. Профиль `strict` их не отфильтрует
(находка имеет confidence `high`); при работе с такими модулями лучше не
полагаться на эту находку.

### Профили

Параметр `profile` (или `default_profile` в конфиге):

- **`full`** (дефолт) — все находки, `level` как передан. Для сильной модели,
  которая сама отбрасывает сомнительное.
- **`strict`** — только high-confidence находки и форсированный `level=1`. Для
  слабых моделей, чтобы ложное срабатывание не приводило к зацикливанию.

## Архитектура

Cargo workspace из пяти крейтов:

| Крейт | Назначение |
|-------|-----------|
| `hbk-reader` | Чтение бинарного контейнера `shcntx_ru.hbk` |
| `hbk-parser` | Парсинг HTML-страниц справки (типы, методы, перечисления) |
| `platform-index` | Индекс платформы: загрузка, хранение, поиск |
| `bsl-validator` | Валидатор BSL-выражений (tree-sitter) |
| `server` | HTTP MCP-сервер (axum + rmcp), конфиг, PID-lock |

## Требования

- Rust (edition 2021), сборка `cargo build --release`.
- Файл `shcntx_ru.hbk` из установленной платформы 1С:Предприятие
  (`C:\Program Files\1cv8\<версия>\bin\shcntx_ru.hbk`). В репозиторий не входит.

## Сборка

```bash
cargo build --release
```

Бинарник — `target/release/bsl-context-rs` (`.exe` на Windows).

## Конфигурация

Скопируйте [`configs/config.toml.example`](configs/config.toml.example) в
`configs/config.toml` и подправьте под свою машину. Ключевые поля:

```toml
host = "127.0.0.1"          # bind, loopback по умолчанию
port = 8007                 # порт MCP-сервера
platform_path = 'C:\Program Files\1cv8\8.3.27.1786'   # каталог версии платформы
default_validation_level = 1
```

### Выбор версии платформы при нескольких установках

Если на машине стоит несколько версий платформы рядом, сервер **не выбирает
версию автоматически** — путь задаётся явно в `platform_path`. Внутри указанного
каталога ищется `shcntx_ru.hbk` по двум путям: `<platform_path>/shcntx_ru.hbk`
и `<platform_path>/bin/shcntx_ru.hbk`.

Это осознанное решение: в разных версиях платформы отличаются сигнатуры методов
и состав системных перечислений, поэтому валидировать код нужно против той версии,
под которую он пишется. Если `platform_path` не задан — сервер стартует, `/health`
отвечает, но MCP-инструменты возвращают `503` с подсказкой указать путь.

### Сетевой деплой

По умолчанию сервер слушает loopback. При `host = "0.0.0.0"` нужно добавить
внешний адрес в `allowed_hosts` (защита rmcp от DNS-rebinding), иначе запросы по
сети получат `403 Forbidden: Host header is not allowed`:

```toml
allowed_hosts = ["localhost", "127.0.0.1", "::1", "<ip-сервера>"]
```

### Внешний источник имён конфигурации

Валидатору присылают текст ОДНОГО модуля, поэтому вызов процедуры, объявленной в
другом файле конфигурации, он по умолчанию считает опиской. Источник имён закрывает
это: он отвечает на три вопроса — «есть ли в конфигурации такой метод», «это
экспортный метод глобального общего модуля» и «какие экспортные методы у модуля
объекта-владельца внешней обработки».

Замер на конфигурации УТ (14905 модулей): без источника — 1420 находок
`undeclared_method` высокой уверенности, с источником — 44, из них настоящая одна.

Три способа подключения, результат одинаковый:

```toml
# 1. Собственный облегчённый индекс. Продукт автономен, ничего внешнего не нужно.
[symbol_source]
kind = "lite"
root = 'C:\RepoUT'                          # каталог выгрузки конфигурации
db_path = 'C:\tools\bsl-context\ut_lite.db' # файл базы; каталог создастся сам
```

Базу собирает инструмент `rebuild_symbol_index` — вызовите его после первого запуска
и после каждой доработки конфигурации. Пути он берёт из конфига. Пока идёт сборка,
валидация продолжает работать по прежней базе; если сборка упадёт, прежняя база
останется целой. То же самое можно сделать из командной строки:
`bsl-lite-index build --root <выгрузка> --db <файл.db>` (каталог должен существовать
заранее).

```toml
# 2. Прямое чтение базы code-index (только если она на этой же машине).
[symbol_source]
kind = "code_index_db"
db_path = 'C:\RepoUT\.code-index\index.db'
```

```toml
# 3. Через запущенный сервис code-index — по адресу и порту, машина любая.
[symbol_source]
kind = "code_index_mcp"
url = "http://127.0.0.1:8011/mcp"
repo = "ut"
timeout_ms = 5000
```

Без секции (или при `kind = "none"`) валидатор работает как раньше, не зная ничего
о конфигурации.

Чем различаются. `lite` не требует ничего постороннего и отвечает из памяти, но его
базу нужно пересобирать после доработок конфигурации. Источники поверх `code-index`
берут данные из его индекса — если тот обновляется наблюдателем за файлами, имена
всегда свежие. Признак «глобальный общий модуль» они читают из XML модуля, который
`code-index` хранит целиком.

По скорости источники неравноценны. Замер на базе в 261 548 функций: прямое чтение
SQLite — 13 мкс на запрос (а после подъёма имён в память — 142 нс), запрос к
`code-index` по сети — 13,4 мс. Валидатор проверяет сотни вызовов на модуль, поэтому
`lite` и `code_index_db` предпочтительнее, а `code_index_mcp` берут тогда, когда
индекс лежит на другой машине.

### Несколько конфигураций на одном сервере

Один сервер обслуживает сразу несколько конфигураций — у каждой свой способ доступа.
Вместо одиночной секции `[symbol_source]` перечислите их списком:

```toml
[[symbol_sources]]
repo = "ut"                    # алиас конфигурации: это значение параметра repo у инструментов
kind = "lite"
root = 'C:\RepoUT'
db_path = 'C:\tools\bsl-context\ut_lite.db'

[[symbol_sources]]
repo = "bp"
kind = "code_index_db"
db_path = 'C:\RepoBP\.code-index\index.db'

[[symbol_sources]]
repo = "zup"
kind = "code_index_mcp"
url = "http://10.0.0.5:8011/mcp"
code_index_repo = "zup-prod"   # необязательно: только если в code-index репозиторий назван иначе
```

Тогда у `validate_module` и `rebuild_symbol_index` появляется параметр `repo` — тот
самый алиас. **Он обязателен, если настроена хотя бы одна конфигурация**, даже
единственная: вызов должен быть однозначным. Промах по имени — отказ со списком
доступных алиасов. Секции `[symbol_source]` и `[[symbol_sources]]` взаимоисключающи.

Если конфигураций не настроено вовсе, параметр не нужен: код проверяется только
против справки платформы.

### Что делает сервер, когда источник имён недоступен

Источник, отвечающий «такого метода нет» на всё, страшнее отсутствия источника: у
валидатора это превращается в находку «метод не объявлен» с высокой уверенностью на
каждый вызов процедуры. Поэтому:

- **При подключении** `code_index_mcp` сервер спрашивает у `code-index` статистику по
  своему репозиторию. Если тот его не знает — источник не создаётся, а в журнал идёт
  ошибка со списком доступных имён.
- **Конфигурация объявлена, но её источник не поднялся** — `validate_module` по этому
  алиасу отказывает и объясняет причину (для `lite` — «индекс не собран, вызовите
  `rebuild_symbol_index`»), а не выдаёт заведомо ложные находки.
- **Источник отвалился на ходу** (сеть, упавший `code-index`) — он помечается
  нерабочим, пустой ответ не попадает в кэш, а валидация по этой конфигурации
  отказывает до восстановления.

### Белый список инструментов

Если от сервера нужна только часть возможностей — перечислите нужные инструменты
в секции `[tools]`. Например, оставить валидацию модуля и две справочные подсказки:

```toml
[tools]
enabled = ["validate_module", "get_constructors", "get_enum_values"]
```

Секция отсутствует или список пуст — доступны все девять инструментов, как раньше.
Скрытые инструменты не попадают в `tools/list` и не выполняются при прямом вызове.
Неизвестное имя в списке сервер не роняет: в лог уходит предупреждение, а сам
инструмент просто не появится.

## Запуск

```bash
bsl-context-rs --config /path/to/config.toml
```

Healthcheck — `GET http://127.0.0.1:8007/health` (без MCP-handshake).

## MCP-инструменты

Транспорт — Streamable HTTP на `http://127.0.0.1:8007/mcp` (stateless).

| Инструмент | Назначение |
|-----------|-----------|
| `search` | Нечёткий поиск по типам, глобальным методам, свойствам |
| `info` | Детали по точному имени |
| `get_member` | Конкретный метод/свойство типа |
| `get_members` | Все члены типа (методы + свойства + значения перечисления) |
| `get_constructors` | Конструкторы типа с сигнатурами |
| `get_enum_values` | Значения системного перечисления |
| `validate_enum` | Проверка значения перечисления |
| `validate_method_call` | Проверка числа аргументов глобальной функции |
| `validate_module` | Валидация BSL-кода (целый модуль или фрагмент) против платформы |
| `rebuild_symbol_index` | Пересобрать собственный индекс имён (`kind = "lite"`); пути — из конфига |

## Подключение к MCP-клиенту

```json
{
  "mcpServers": {
    "bsl-context": {
      "type": "http",
      "url": "http://127.0.0.1:8007/mcp"
    }
  }
}
```
